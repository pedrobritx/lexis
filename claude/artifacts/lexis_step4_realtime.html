Part 1 — Lock-based conflict resolution
When a user begins editing an object, it becomes exclusively theirs until they release. Other users see a visual lock indicator and cannot move or edit the object. The simplest model that eliminates conflicts entirely — appropriate for 1-on-1 and small group teaching.

Lock acquisition: client emits object:lock when pointer-down on an object or when entering text edit mode. RT server checks Redis key lock:{objectId} — if absent, sets it with TTL = 30s and the locking user's ID, broadcasts object:locked to all room members. If already set by another user, returns object:lock_rejected to requester only.
Lock TTL safety: the 30-second TTL ensures locks are auto-released if the user disconnects or the tab crashes. On every pointermove or input event while holding a lock, the client emits object:lock_keepalive which resets the TTL. A user actively editing never loses their lock mid-operation.
Lock release: client emits object:unlock on pointer-up, click-away, or blur. RT server deletes the Redis key, broadcasts object:unlocked with the final object state. The unlock event carries the committed delta — other clients apply it immediately.
Visual indicator: locked objects show the locking user's avatar and name in a small badge at the top-left corner. The object border pulses with the user's assigned colour (each presence user gets a colour from a fixed palette of 6). Move and resize handles are hidden. Text cursor shows a "blocked" cursor CSS on hover.
Teacher override: teacher role can emit object:force_unlock to break any lock in their tenant's board. Used when a student's tab crashes and leaves a stale lock before the TTL expires. Broadcasts object:unlocked with no delta — object state unchanged, lock cleared.
Part 2 — Shared undo/redo
A single command log shared across all users in a board page. Undo reverses the most recent action regardless of who performed it. This matches the collaborative model — everyone is building the board together, and mistakes should be reversible by anyone.
Command log architecture
Every mutation is a reversible command stored server-side
Command log: every mutable operation on the board is stored in board_commands table: command_type (object_create / object_update / object_delete / stroke_add / stroke_erase), forward_payload (what was applied), reverse_payload (what reverses it), author_id, sequence_num (monotonically increasing per page), undone_at nullable timestamp.
Undo: client emits board:undo. RT server finds the most recent command for this page where undone_at IS NULL, applies its reverse_payload, sets undone_at = now(), broadcasts object:update or object:delete with the reversed state to all clients. The undo stack is the command log itself — no separate data structure needed.
Redo: client emits board:redo. RT server finds the most recent command where undone_at IS NOT NULL (the last undone action), re-applies its forward_payload, clears undone_at, broadcasts the re-applied state.
Stroke undo: strokes live in S3, not PostgreSQL — they can't be individually reversed by a DB update. The command log stores the strokeId in reverse_payload. Undoing a stroke emits a stroke:hide event which adds the ID to a Redis set hidden_strokes:{pageId}. Clients filter hidden stroke IDs when rendering. On S3 flush, hidden strokes are excluded from the binary blob permanently.
History depth: the command log is kept for 90 days, capped at 500 commands per page. Beyond 500, the oldest commands are pruned and their effects made permanent. Visual undo history is shown as a scrollable timeline in a board settings panel — each entry shows the action, the author's avatar, and the timestamp.
Part 3 — Zoom overlay system
PDFs and activity cards expand to a fullscreen overlay when zoomed. The canvas stays live underneath. Drawing tools remain fully active over the overlay — the teacher can annotate a PDF while the canvas with other objects is still visible at the edges.
Layer architecture — overlay model
5-layer z-stack · Drawing tools always topmost

Drawing tools HUD
Toolbar, colour picker, width selector — always on top of everything. Never obscured by overlay.
z:40

Transparent canvas layer. Active when overlay is open. Strokes drawn here belong to the PDF/activity object, not the board. Emitted as annotation:stroke:delta.
z:30

PDF.js renderer or activity card UI. Fills 90% of viewport with a 5% margin. Semi-transparent backdrop dims the canvas underneath (rgba 0,0,0,0.6).
z:20

All canvas objects (stickies, shapes, text). Visible at edges when overlay is open at less than 100% width.
z:10
Board stroke canvas
Freehand strokes on the board. Paused during overlay — pointer events captured by annotation canvas instead.
z:0
Pattern background
Blank / dotted / squared CSS pattern.
Opening the overlay: user clicks the zoom button on any PDF card or activity card → overlay:open event emitted → RT server broadcasts to all room members → all clients open the overlay simultaneously showing the same object. The canvas underneath stays interactive for participants not in follow mode.
Drawing over PDF: when overlay is open, pointer events are captured by the annotation canvas (z:40) instead of the board stroke canvas (z:10). Strokes drawn here emit annotation:stroke:delta events tagged with the objectId of the PDF. They render on the annotation canvas and sync to all clients in real time via the same stroke pipeline.
Drawing over activity: same model — annotation canvas captures strokes. Teacher can circle a student's incorrect answer, draw an arrow to the correct option. These annotations are stored per activity attempt — they appear when the teacher reviews the submission, creating a visual correction history.
Closing the overlay: Esc key, click outside overlay, or close button. Emits overlay:close → all clients close. Annotation strokes drawn during the session are flushed to S3 tagged to the object. Board stroke canvas resumes capturing pointer events.
Follow mode + overlay: when follow mode is active and teacher opens the overlay, all students automatically see the same overlay open on the same object. If teacher closes the overlay, all students close. Follow mode extends into the overlay layer seamlessly.
Part 4 — PDF annotation system
PDF objects on the canvas are first-class annotatable documents. Annotations are shared — teacher and student see each other's marks in real time. Three annotation types: freehand strokes, text comments, and highlights. All anchored to PDF coordinates, not screen coordinates.
Annotation coordinate system — PDF-space, not screen-space
Annotations survive zoom, scroll, and resize
PDF.js renders pages at a given scale factor. All annotation coordinates are stored in PDF-space — normalised to the page dimensions (0.0–1.0 on both axes). On render, the client transforms from PDF-space to screen-space using the current PDF.js scale. This means annotations remain perfectly aligned regardless of zoom level, screen size, or device. A highlight drawn at PDF coordinates (0.1, 0.4) → (0.6, 0.45) renders correctly on a 13" MacBook at 100% zoom and on an iPad at 150% zoom.
Three annotation types
Freehand · Text comment · Highlight
Freehand annotation: uses the drawing tools HUD (pen, highlighter, eraser). Strokes drawn over the overlay PDF are tagged with target_type: 'pdf', target_id: objectId, page_num: N. Points stored in PDF-space coordinates. Synced via annotation:stroke:delta in real time. Persisted to pdf_annotations table on stroke end, separate from the board stroke S3 binary.
Text comment: double-tap anywhere on the PDF page opens an inline text input anchored to that PDF-space coordinate. Stored as annotation_type: 'comment' with position: {x, y} and content: text. Rendered as a small speech bubble floating at that position. Visible to all. Click to expand, click away to collapse. Author avatar shown on the bubble.
Highlight: select text in PDF.js (drag over text), "Highlight" button appears in a floating toolbar. Stores the selection as annotation_type: 'highlight' with PDF-space bounding box and optionally a comment. Highlights are rendered as a coloured overlay rect on the annotation canvas — teacher highlights in amber, student highlights in sage.
Teacher pushes a page: teacher emits pdf:push_page with {objectId, pageNum}. RT server broadcasts to all clients in room. All students' PDF viewer scrolls to that page. Combined with follow mode — when follow is active, page changes automatically sync. Without follow mode, the push is a one-time nudge that the student can override.
Independent annotations: even though annotations are shared, the student has their own independent copy of the PDF object via pdf_annotation_sets — a personal layer. Teacher's annotation set is visible to the student as read-only coloured marks. Student's set is visible to teacher. This creates a visual dialogue on the document without either overwriting the other.
Part 5 — WebSocket hardening
Two hardening features: connection health indicator (latency badge, reconnect status) and automatic reconnection with zero data loss. A dropped connection should be invisible to both teacher and student.
Connection health indicator
Always visible · Never alarming unless actionable
Latency measurement: client sends a ping every 5 seconds with a timestamp. Server responds with pong carrying the same timestamp. Client computes round-trip time. Displayed as a small indicator in the session HUD: green (<100ms), amber (100–300ms), red (>300ms or timeout). Shows exact ms on hover.
Reconnecting state: when Socket.IO detects a disconnect, the HUD indicator switches to a pulsing "Reconnecting…" badge. The board is still fully visible and readable — only new events are paused. The client continues to accept local edits and queues them in memory (not in localStorage — in-memory only, cleared on hard refresh).
Connection quality detail: clicking the health indicator opens a small popover showing: current latency, last 10 measurements as a sparkline, packet loss percentage, and the Socket.IO transport type (websocket / polling fallback). Useful for diagnosing poor session quality.
Automatic reconnection with state recovery
Sequence numbers ensure no event is lost or replayed twice
Sequence numbers: every event the RT server broadcasts is stamped with a monotonically increasing seq integer per board page, stored in Redis as seq:{pageId}. Clients track the highest seq they have received as lastSeq.
Reconnection handshake: Socket.IO's built-in reconnection fires automatically. On reconnect, the client sends board:join with {pageId, token, lastSeq}. The server checks Redis for events since lastSeq in a replay buffer (last 500 events per page, 5-minute TTL). Missing events are replayed to the reconnecting client only. No duplicate events — seq guards against re-processing.
Replay buffer: a Redis list replay:{pageId} stores the last 500 serialised events with their seq. New events are prepended (LPUSH), old events trimmed (LTRIM to 500). On reconnect with a lastSeq that's within the buffer, only missing events are replayed. If lastSeq is older than the buffer, a full board state refresh is sent instead (same as initial join).
In-memory queued edits: edits made during the disconnection window are queued in memory. On reconnect and after the replay completes, queued edits are applied on top of the recovered state and emitted to the server. The result is seamless — the user sees their edits persist as if the connection never dropped.
