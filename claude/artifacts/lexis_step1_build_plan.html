Week 2 Placement test + curriculum backbone Backend Days 6–10
Day 6 Placement test — delivery Backend 7 hrs
Write placement test question bank as a seed file — 12 MCQ questions spanning A1→C1 grammar
Implement GET /v1/placement/test — return randomised question set (no answers in response)
Implement POST /v1/placement/submit — receive answers, score against answer key, compute CEFR level
CEFR scoring logic: A1 ≤3, A2 4–5, B1 6–7, B2 8–9, C1 10–11, C2 12 correct
Write result back to PlacementTest table and update StudentProfile.cefr_level
Handle skip path — POST /v1/placement/skip with manual level assignment (teacher role only)
Deliverable Placement test delivers, scores, and writes CEFR level to student profile
Day 7 Courses module — CRUD + versioning Backend 7 hrs
Implement full CRUD for Course: POST, GET, PATCH, DELETE /v1/courses
Implement Unit CRUD nested under course: /v1/courses/:id/units
Implement Lesson CRUD nested under unit: /v1/units/:id/lessons
Status machine: draft → active → archived with guards (can't archive if classroom enrolled)
Auto-increment version on any update to course, unit, or lesson
Teacher-only write access, read access for enrolled students
Deliverable Teacher can create full course hierarchy, versioning increments on edits
Day 8 Activities module — type registry Backend 7 hrs
Implement Activity CRUD: /v1/lessons/:id/activities
Build activity type registry — each type (cloze, mcq, matching, ordering, open) registers a content schema + validator
Implement answer validation endpoint: POST /v1/activities/:id/validate — returns correct: bool, score: float, feedback: string
Cloze validator: normalise whitespace + case, accept near-matches flag
MCQ validator: single correct option check
Matching validator: all pairs must match, partial scoring supported
Write validator unit tests for each type with correct and incorrect inputs
Deliverable All 4 activity types validate correctly, type registry extensible with one file per type

Day 11 Lesson engine — sequencing + hints Backend 7 hrs

Implement GET /v1/lessons/:id/next-activity — returns next unattempted activity for current progress

Deliverable Engine correctly sequences, handles hints, retries, and produces lesson summaries
Day 12 SRS module — SM-2 implementation Backend 7 hrs
Implement SM-2 algorithm in srs/sm2.ts — inputs: quality (0–5), ease_factor, interval, repetitions → outputs: new interval, new ease_factor
Listen to activity.attempted event — create or update SrsItem for each activity based on correctness
Implement GET /v1/srs/queue — return all SrsItem records where next_review ≤ today, ordered by urgency
Implement POST /v1/srs/review/:itemId — record quality, run SM-2, update SrsItem, log to SrsLog
Streak logic: update StudentProfile.streak_days — increment if reviewed today, reset if >1 day gap
Unit tests for SM-2: quality 0 resets interval, quality 5 grows interval correctly
Deliverable SM-2 schedules items correctly, review queue populated after lesson completion
Day 13 Gamification — XP, badges, streaks Backend 6 hrs

Badge award logic: check trigger_criteria JSON against student state, insert StudentBadge if criteria met
Implement GET /v1/students/:id/badges — return earned + locked badges
Implement GET /v1/students/:id/xp — XP total, level, progress to next level
Emit badge.earned event (for future push notifications)
Deliverable Badges awarded automatically on trigger events, XP level computed correctly
