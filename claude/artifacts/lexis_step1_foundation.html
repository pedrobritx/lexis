Part 1 — Architectural decisions
All decisions made before writing the first migration. These shape every table, every API route, and every client.
Authentication — Passkeys + magic link, no passwords
WebAuthn (Apple/Google) · Email OTP fallback · Zero credential storage
How it works: Users register with their Apple ID, Google account, or an email address. No password is ever created or stored. Authentication happens via two paths:
Passkey (primary): Uses the WebAuthn API — the device biometric (Face ID, Touch ID, Windows Hello) signs a challenge. The server verifies the signature against the stored public key. The private key never leaves the device. Works natively on iOS via ASAuthorizationController, on Android via Credential Manager, and on web via navigator.credentials.
Magic link (fallback): User enters their email → server generates a 6-digit OTP stored in Redis with a 10-minute TTL → email sent via Resend → user enters the code → server issues JWT. No link to click, just a code. Works on any device without a passkey.
JWT issuance: After either auth path succeeds, the server issues a short-lived access token (15 min) and a long-lived refresh token (30 days, stored in Redis, rotated on use). Same JWT flow as before — auth method changes, token mechanics stay identical.
Data model change: hashed_password removed from users. New table: passkey_credentials (stores credential_id, public_key, sign_count, device label). One user can have multiple passkeys (phone + laptop + tablet).
Multi-tenancy — shared DB, tenant_id on every mutable table
One PostgreSQL instance · Row-level isolation · tenant_id from JWT only
The rule: every table that holds teacher or student data has a tenant_id UUID FK referencing the tenants table. Every API query appends WHERE tenant_id = req.user.tenantId. The tenantId is extracted from the JWT — never from the request body or query parameters. A client cannot forge their tenant context.
Tenant creation: when a teacher registers, a tenants record is auto-created with a free-tier subscription. The teacher's user record gets tenant_id set. Students are invited into a tenant by their teacher — they never create tenants.
Cross-tenant read prevention: a Prisma middleware layer appends tenant_id to every query automatically. If a route handler forgets to scope by tenant, the middleware catches it. This is the primary security layer.
Tables exempt from tenant_id: users (auth only), passkey_credentials (tied to user, not tenant), magic_link_tokens (ephemeral), badges catalogue (global), and public templates (see content policy below).
Billing — Freemium with feature flags · Stripe
Free: 3 students, 5 lesson plans · Pro: unlimited · Flags control all gates
Free tier limits: a teacher can enroll up to 3 students and create up to 5 lesson plans (courses). All other features — activities, classrooms, spaced review, sessions — are unrestricted on free. The limits are enforced by checking subscription.student_limit and subscription.lesson_plan_limit before the relevant create operations.
Feature flags: subscriptions.feature_flags: jsonb controls access to future features (AI generation, analytics dashboard, whiteboard recording, priority support). Adding a new gated feature means adding one key to the flags JSON — never touching application code for existing tiers.
Enforcement point: limit checks live in the enrollments module (POST /v1/classrooms/:id/enroll) and the courses module (POST /v1/courses). Both return 402 Payment Required with a structured error when a limit is hit, so the client can show an upgrade prompt.
Stripe integration: tenants.stripe_customer_id is null on free. When a teacher upgrades, a Stripe customer is created, a subscription is attached, and a webhook updates subscriptions.plan_slug and raises the limits. The webhook is the only place that can change a subscription — never a direct API call from the client.
Content visibility — private by default, public templates opt-in
visibility enum · Templates readable by all · Writable by owner only
Visibility model: Course, Lesson, and Activity each get a visibility enum: private (default, tenant-only) or public_template (readable by any authenticated user, writable only by the owning tenant). A teacher can mark their course as a public template, making it discoverable in a shared template library.
Template reads bypass tenant_id filter: GET /v1/templates queries courses where visibility = 'public_template' with no tenant scoping. A teacher can browse and clone any public template into their own tenant — cloning copies all units, lessons, and activities and sets tenant_id to the cloner's tenant.
Write protection: even though a template is publicly readable, all mutating operations (PATCH, DELETE) still check tenant_id. You can read anyone's template, clone it, but never edit the original.
Lexis-curated templates: a special system tenant (seeded at setup) owns the official CEFR-aligned templates. No teacher can edit system templates. This is the content seed that replaces the "no content at launch" critical issue.
