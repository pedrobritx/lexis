Part 1 — Language-agnostic architecture
Lexis starts as an English platform. But the infrastructure must never assume English. Every language-specific element is parameterised from day one so that adding Spanish, French, or Mandarin requires zero schema migrations.
What changes when you go language-agnostic
Three fields, one enum, one index — that's it
Making Lexis language-agnostic is not a rewrite — it's a parameterisation. The teaching logic, activity types, SRS algorithm, and whiteboard are all language-independent already. The only language-specific elements are the content (courses, lessons, activities) and the proficiency framework (CEFR for European languages, JLPT for Japanese, HSK for Mandarin, etc.).
Add target_language to courses: ISO 639-1 code (en, es, fr, zh, ja, etc.). Default en. This is the language being taught — not the teacher's language. A Portuguese teacher teaching Spanish sets target_language = 'es'.
Add framework to courses: enum cefr | jlpt | hsk | custom. Default cefr. This controls which level system the course uses. JLPT goes N5→N1, HSK goes 1→6, custom allows the teacher to define their own levels. The target_cefr field is renamed target_level and stores the level within whichever framework is active.
Add teacher_language to teacher_profiles: the language the teacher uses to explain — used to localise the UI and any AI-generated grammar explanations. Separate from the language being taught.
The placement test becomes framework-aware: the CEFR question bank is one implementation. Adding JLPT means adding a second question bank tagged framework = 'jlpt'. The placement module routes to the correct bank based on the course's framework. No architectural change — just content addition.
Public templates are language-tagged: the template library is filterable by target_language and framework. A teacher looking for Spanish B2 templates sees only Spanish CEFR content — not English or Mandarin.

Why manual approval for Pro
Quality signal, not friction
Pro and Growth require a brief manual review — not to create friction, but to ensure quality. A spam teacher with 50 fake students breaks the platform's reputation. The review is lightweight: teacher submits a short profile (teaching subjects, experience, language), you approve within 24 hours. This also gives you a direct relationship with every paying teacher early on — invaluable for product feedback. Self-serve Pro can be enabled later when you have moderation tooling.

Upgrade request flow — implementation detail
Simple admin email, no custom UI needed in Phase 2
The upgrade request is a POST /v1/billing/upgrade-request endpoint. It creates a upgrade_requests record and sends you (the admin) an email via Resend with the teacher's profile and a one-click approve link. Clicking approve calls POST /v1/admin/upgrade-requests/:id/approve which sets subscription.status = 'pending_payment' and sends the teacher a Stripe checkout link. No custom admin UI needed in Phase 2 — just an email + a URL. Build the admin dashboard in Phase 4.
