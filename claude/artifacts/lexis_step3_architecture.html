Part 1 — Real-time transport architecture
Socket.IO on a dedicated Node.js process, Redis pub/sub as the message broker. The API server handles persistence; the RT server handles broadcast. They never talk to each other directly — only via Redis channels.

How a drawing stroke flows through the system
From pen-down to all screens in <50ms
1. Client draws: pointer events fire → stroke points buffered locally at 60fps → emitted to RT server as stroke:delta events every 16ms (one frame). Never sent to the API server directly.
2. RT server broadcasts: receives stroke:delta → publishes to Redis channel board:{pageId} → all other connected clients in that room receive it via Socket.IO room broadcast. Latency: <50ms on same region.
3. Stroke persistence: RT server accumulates stroke deltas in Redis key strokes:{pageId}:buffer. A BullMQ job flushes every 60 seconds — serialises buffer to binary (MessagePack), uploads to S3 at strokes/{pageId}/{timestamp}.bin, clears Redis buffer, updates board_strokes.strokes_url.
4. Board load: client requests GET /v1/board/pages/:id from REST API → receives structured objects (stickies, activity cards, PDFs) from PostgreSQL + a signed S3 URL for the stroke binary. Client renders both layers independently.
5. Async access (homework): outside live sessions, clients connect to the RT server in async mode — same socket, same room, but no presence indicators. Changes persist identically. The board is always live; the session is what has presence.

Part 2 — Whiteboard engine features
Every feature specified, with implementation detail. All built in Phase 2. Canvas rendering uses the HTML5 Canvas API on web and Core Graphics on iOS.
Drawing tools
Pen (pressure-sensitive via Pointer Events API), highlighter (semi-transparent overlay, 40% opacity), eraser (stroke-based — removes intersecting strokes, not pixel-based). Each tool has color, width, and opacity properties stored per stroke in the event payload. Tool state is local — never synced, only the output strokes are.
Shapes & connectors
Rectangle, ellipse, line, arrow, diamond. All stored as BoardObject with object_type = 'shape' and geometry in content.geometry. Connectors snap to object anchor points — 4 cardinal points per shape. Connection is stored as {from: objectId, to: objectId, anchor: 'top'|'right'|'bottom'|'left'}. When a connected object moves, the connector recomputes its path client-side.
Object locking
Any object can be locked via context menu or toolbar. Locked objects have locked: true in their content. The RT server rejects object:update and object:delete events for locked objects. Writing inside a locked text box or activity card is still permitted — only position and size are frozen. Visual indicator: a small lock icon in the top-left corner of the object.
Magnetic alignment + rulers
When dragging an object, snap guides appear at: edges of other objects (±8px threshold), canvas centre lines, and ruler intervals. Ruler spacing is configurable (10px, 20px, 50px). Snap is client-side only — no server involvement. On release, the snapped position is sent as the final object:update. Shift key temporarily disables snap. Rulers are rendered as a fixed overlay layer above the canvas grid, below all objects.
Follow mode
Teacher presses "Follow" button in session HUD. Emits follow:start with current viewport state. Student's canvas is locked to teacher's viewport — pan and zoom are disabled. Teacher's subsequent pan/zoom emits follow:viewport events; student canvas mirrors them. Student sees a "Following Pedro" banner with a "Break free" button that sends follow:break — teacher is notified. Teacher ends follow mode with the same button, emitting follow:end.
Board patterns
Three patterns: blank (solid colour), dotted (radial gradient dots, default), squared (grid lines). Pattern preference stored in board_pages.pattern enum: blank|dotted|squared. Rendered as the bottom CSS layer — a background-image pattern on the canvas wrapper div. Switching is instant and persists per page. Each teacher can set a default pattern in their account settings.
Laser pointer
Teacher activates via toolbar or keyboard shortcut (L). While active, mouse/pen position emits laser:move events continuously. Student canvas renders an amber circle at those coordinates with a short fade trail (CSS animation, 300ms). No persistence — ephemeral only. Deactivating emits laser:end which removes the indicator. Laser state is not shown on the teacher's own screen — only on students'.
Activity cards as canvas objects
Activity cards are BoardObject with object_type = 'activity' and content.activity_id. The card renders in two states: collapsed (chip showing type + title) and expanded (full interactive activity). Expanding calls GET /v1/activities/:id from the REST API — the RT server is never involved in activity content. Submitted answers call POST /v1/progress/activities/:id/attempt normally. The card can be locked to prevent accidental movement during active completion.
Part 3 — Assessment system
Four assessment types in Phase 2. Auto-graded types extend the existing activity validator registry. Teacher-graded and listening types introduce new submission and review flows.
Auto-graded types — extended
MCQ, cloze, matching, ordering — Phase 1 validators plus ordering
Ordering: student drags items into correct sequence. content.items contains the shuffled list; answer_key.order contains correct indices. Validator compares index arrays — partial scoring gives credit for each correctly placed item relative to its neighbours.
Matching: two columns, drag left items to match right items. answer_key.pairs is an array of [leftIndex, rightIndex] tuples. Partial scoring: each correct pair = 1/N of total score.
Near-match cloze: add a scoring_rules.accept_near_match: boolean field. When true, the validator uses Levenshtein distance ≤2 for single-word answers to accept minor spelling errors without marking wrong.
Teacher-graded open writing
Rubric optional at creation, always editable at grading
Submission flow: student types response in a rich text area → submits → ActivityAttempt created with status = 'pending_review' and response.text. Teacher is notified via event bus (submission.pending).
Rubric structure: stored in activities.scoring_rules.rubric as an array of criteria: [{criterion: "Grammar accuracy", max_points: 5}, {criterion: "Vocabulary range", max_points: 5}]. Optional — if null, teacher grades with a single score and comments only.
Grading flow: teacher opens GET /v1/submissions/pending → sees all unreviewed submissions → selects one → edits rubric criteria scores inline → adds text annotations → submits via POST /v1/submissions/:id/grade. Student is notified via event bus (submission.graded).
Rubric editability: even if a rubric was pre-defined on the activity, the teacher can adjust criteria scores and add/remove criteria at grading time. The final grade is what was submitted at review, not the pre-defined rubric.
Listening assessment — audio prompt + audio response
Student hears audio, records spoken response in-app
Activity structure: type = 'listening'. content.audio_asset_id references a media_assets record. content.prompt_text is an optional written prompt shown below the audio player. content.response_type is 'audio' (always in Phase 2 — typed-only option is Phase 3).
Recording flow: student presses record → browser MediaRecorder API captures microphone → audio chunked into WebM/Opus blobs → on stop, blobs assembled → uploaded to S3 via POST /v1/media/upload → ActivityAttempt.response.audio_asset_id set → attempt submitted with status = 'pending_review'.
Teacher review: same grading flow as open writing. Teacher plays the student's recording inline, scores with rubric or single score, adds text feedback. Phase 4 will add ASR auto-scoring as an AI layer on top of this flow — the data model already supports it.
iOS: uses AVAudioRecorder to capture audio. Uploads the M4A file to the same POST /v1/media/upload endpoint. The server normalises to WebM/Opus via FFmpeg before storing — client format is irrelevant to the API.
Part 4 — Media pipeline
Five asset types. All uploads go through a single POST /v1/media/upload endpoint that validates, processes, and stores. Assets are never served directly from S3 — always via signed URLs with short TTLs.
Upload pipeline — every asset type
Single endpoint, type-aware processing, storage-limit checked before accept
Images (JPEG/PNG/WebP): client uploads → server checks storage limit → Sharp resizes to max 2048px on longest side, strips EXIF, converts to WebP → stores to S3 at media/{tenantId}/{uuid}.webp. Thumbnail generated at 400px for previews. media_assets record created with asset_type = 'image'.
PDFs: stored as-is after virus scan (ClamAV or VirusTotal API). First page rendered to WebP thumbnail via pdf-thumbnail. asset_type = 'pdf'. Client renders using PDF.js — S3 signed URL passed directly. No server-side PDF rendering during lesson delivery.
Audio (uploaded): FFmpeg normalises to WebM/Opus regardless of input format (MP3, M4A, WAV). Waveform data extracted and stored in metadata.waveform jsonb for client visualisation. asset_type = 'audio'.
Audio (teacher records in-app): MediaRecorder API captures directly as WebM/Opus — no transcoding needed. Uploaded via the same endpoint. Indistinguishable from uploaded audio after storage.
Video embeds (YouTube/Vimeo): not uploaded — stored as asset_type = 'video_embed' with metadata.embed_url and metadata.provider. The server fetches the oEmbed endpoint to validate the URL and store the thumbnail. No video bytes ever touch Lexis servers. Counts zero bytes against storage quota.
Signed URL delivery: GET /v1/media/:id/url returns a signed S3 URL valid for 1 hour. Clients never store permanent S3 URLs — they request fresh signed URLs on each render. This means you can revoke access to any asset by soft-deleting the media_assets record.
Part 5 — Spaced review delivery modes
Teacher assigns review mode per SRS item. Two modes available. Mode stored on the SRS item, not globally. Students see a mixed queue rendered according to each item's assigned mode.
Flashcard mode
One item at a time. Front shows the prompt (e.g. grammar structure, vocabulary word, sentence fragment). Student taps to reveal the answer. Then rates quality: Again (0) / Hard (2) / Good (4) / Easy (5) — mapped to SM-2 quality values. Swipe gestures available on mobile. Item moves to next in queue. No correct/incorrect — purely self-assessed. Best for vocabulary and grammar rules.
Mini-lesson mode
3–5 items grouped into a short auto-graded exercise. Same activity types as lessons (cloze, MCQ). Automatically graded — SM-2 quality derived from score: 100% = quality 5, 80%+ = quality 4, 60%+ = quality 3, below = quality 1. Student gets a summary at the end showing which items need more work. Best for grammar forms and structured practice.
How the teacher assigns review mode
Per activity, at creation time or after
When creating or editing an activity, the teacher sees a "Spaced review" toggle with a mode selector: Flashcard or Mini-lesson. This sets activities.srs_mode enum. When a student completes the activity and it enters their SRS queue, the srs_items record inherits srs_mode from the activity at creation time. Teachers can change the mode on the activity — existing SRS items are updated on next review load via the same version-check mechanism as content changes.
